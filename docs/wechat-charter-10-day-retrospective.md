# 10 天、123 次提交：我为什么做了一款「不替代 Claude Code」的 Agent IDE

> 我真正想做的，不是再造一个更聪明的 Agent，而是给那些已经足够强的 Agent，造一间看得见、接得住、验得过的工作室。

前几天，我重新翻了一遍电脑里的开发记录。

从 7 月 12 日晚上到 7 月 22 日，和这个项目直接相关的记录里，有 374 条 Claude Code 输入，分布在 52 个 session ID 中，其中 45 个是有实际开发内容的会话；Codex 里还有 24 个明确以这个仓库为工作目录的会话。

代码仓库里，则留下了 123 次提交、46 份 ADR（架构决策记录），以及两个公开 Beta 版本。

数字看起来有点夸张。但真正让我想写这篇文章的，并不是「十天做了多少功能」，而是我回看这些会话后发现：这个产品几乎不是被一次性设计出来的，而是被我在真实使用中，一次次否定、推翻、重做出来的。

它叫 Charter，一款本地优先的桌面 Agent IDE。

但一开始，我甚至没有真正想清楚它应该是什么。

## 第一个晚上：我把一包规格交给了 Claude Code

7 月 12 日晚上，我对 Claude Code 说：

> 接下来我要去睡觉了。你按照 docs 里的内容实现这个产品，其他决策由你来做。等我明天醒来，我要看到一个完整、可用的产品。

这句话现在看起来很像 AI 编程时代的标准开场：人去睡觉，Agent 通宵干活。

第二天醒来，东西确实跑起来了。项目、文件、终端、Git、Agent、权限、任务状态，基本骨架都有了。测试台账里，当时已经有 201 个单元与集成测试、22 个 Electron 端到端测试。

从「有没有」的角度看，它已经相当完整。

但我打开应用，很快意识到一个更麻烦的问题：它不是我想要的东西。

当时的界面仍然是一套很典型的 IDE 逻辑：文件树、编辑器、终端、Agent 面板。Agent 像是被塞进传统 IDE 侧边栏里的一个新功能。技术上能工作，产品上却没有回答一个最基本的问题：

**如果 Agent 已经可以自己读代码、改文件、跑命令，人为什么还要一直坐在编辑器里？**

这成了整个产品第一次真正的转向。

## 我不想重做 Claude Code，我想保留它

那几天，我反复在 Claude Code 和 Codex 之间讨论一个问题：这个产品到底应该自己实现 Agent，还是直接使用 Pi、Claude Code、Codex 这些现成能力？

我后来越来越确定，后者才是我真正想做的事。

Cursor 这类产品把 IDE 和自研 Agent 做在了一起，体验很完整；Claude Code 和 Codex 的 Agent 能力很强，但主要生活在 CLI 里。CLI 灵活、原生、更新快，却也有天然的缺口：多个 session 很难统一管理，Agent 干了半小时后不容易快速理解过程，预览、审查、回滚、上下文投喂也分散在不同工具里。

所以我的目标，不该是把这些 Agent 换掉，而应该是：

**保留 Claude Code、Codex、Pi 各自原生的能力，再给它们补上一套桌面工作流。**

这也是 Charter 最重要的一条产品原则：Runtime 是可替换的，工作过程和人的控制权才是稳定的产品层。

于是，最早那个「传统 IDE 加 Agent 面板」的方向被推翻了。首页变成一个任务启动器；IDE 工作区被降级为按需打开的 Editor；每个任务拥有自己的 Room；再往后，Task 又逐渐被重新理解为 Session——不是一张一次性工单，而是一段可以持续追问、继续工作、回看和恢复的人机协作关系。

这条路并不直。

Home 和 Editor 曾经是两个割裂的世界；Task Room 曾经信息太多；Workspace 这个入口曾经既代表项目目录，又代表 IDE 页面；回放也一度只是「把文件改动按时间播放一遍」。

我在会话里说得最多的一句话，大概是：

> 不要迎合我。站在产品经理和真正使用者的角度看，这个设计到底有没有问题？

很多时候，答案是：有，而且不是换个颜色、挪个按钮能解决的问题。

## 我真正缺的，不是更强的执行，而是过程可见

Agent 编程最奇怪的地方是：执行能力越来越强，人却越来越容易失去现场感。

以前我自己写代码，知道刚刚改了什么、为什么改、哪条命令失败了。把任务交给 Agent 后，我可能只离开二十分钟，回来就面对几十个文件改动、一段总结，以及一句「已完成」。

总结可以写得很好，但总结不是证据。

这让我逐渐确认，Charter 真正要解决的不是「如何发出 Prompt」，而是按下回车之后的四件事：

1. Agent 正在做什么；
2. 它在哪里做；
3. 结果是否真的有效；
4. 我是否愿意保留这些改动。

围绕这四个问题，产品才开始长出自己的形状。

Agent 写文件时，项目卡片会出现实时活动；某个 session 需要关注或刚刚完成时，左侧会有状态和动效，系统通知可以把我直接带回现场；终端不再只是一整屏滚动文本，而是按命令划分成可定位的语义块；Preview、Console、Diff、Checks 和 Review 被放回同一个 Session 里。

回放也被重新做了几次。最初我以为它应该像视频一样拖时间轴，后来发现这只是把「信息很多」变成「信息按时间很多」。真正有用的回放，应该同时提供三种深度：

- 几十秒看懂发生了什么；
- 沿着关键节点展开原因和上下文；
- 最后回到原始命令、文件、Diff、验证结果这些可核对的证据。

而且，Charter 的用户不一定都在写代码。Agent 可能在整理文档、分析资料、操作其他工具。因此，回放不能只围绕「改了哪些文件」，而要围绕「Agent 为了完成目标，做了哪些可解释的动作」。

这也是我后来非常喜欢的一句话：

**从 Prompt 到 Proof。**

## 终端没有消失，它变成了第一等公民

做这个产品时，我没有把终端当成需要被 GUI 淘汰的旧东西。

相反，我越来越觉得，终端是 Agent 最自然的运行容器。Claude Code 和 Codex 原生的交互、权限、更新节奏，都应该被保留。问题不在终端本身，而在终端之外缺少组织能力。

所以 Charter 会识别真实运行中的 Claude Code 和 Codex，把它们提升为一等 Session：知道它属于哪个项目，保存会话身份，记录开始与结束，支持恢复，把仓库改动带进统一的 Review，同时保留原生 TUI，而不是把它们重新包装成一个假的聊天窗口。

为了验证这件事，我们没有只用 Mock。真实的 Claude Code、真实的 Codex、真实的 PTY、不同安装方式、退出与恢复、终端切换、应用重启，都进过 Electron E2E 流程。

这里也踩过不少很「本机」的坑：macOS 会杀掉被错误复制的系统二进制；同目录并发 session 会让仅靠时间窗口猜测 session ID 的方案失效；终端重新挂载会丢状态；一条默认 `reveal: true` 的调用，会在启动开发服务器时突然把用户从 Room 拽到 Editor。

这些问题在架构图里看不出来，只有真的把产品打开、点进去、跑一遍，才会出现。

## 从一个 Agent，到一支可见的编队

到了 7 月 21 日，我开始讨论一个更进一步的场景。

我经常先和 Claude Code 讨论方案，再把方案交给 Codex 做审查，然后把两边意见来回搬运。既然这些 session 都已经运行在同一个桌面应用里，为什么还要靠我手工复制粘贴？

这就是 Session Orchestration 的起点。

现在，一个主 Session 可以创建、读取、等待和指挥其他可见的 worker terminal。Claude Code 可以启动一个 Codex worker 去审查方案，也可以让多个 worker 分别尝试 A、B、C 三种实现，再等其中一个跑绿。

但我不想做后台黑盒式的 subagent。

每个 worker 都必须是一个真实、可见、可打开的 Session。用户能看到它的输出，能暂停远程控制，能直接敲键盘接管；本地输入永远优先，Agent 发来的内容会排队等待。所有跨终端动作最终经过同一套权限引擎和证据账本，而不是靠一份写给模型看的 Skill 文档维持安全。

这件事让我第一次感到，Charter 不再只是「给单个 Agent 加 UI」，而开始变成一间真正的智能体工作室：人在里面不是监工，也不是完全放手的旁观者，更像一个随时可以切镜、追问、接管和拍板的导演。

## 这十天最重要的一次操作，是撤销

高强度和 AI 一起做产品，很容易产生一种错觉：只要生成得足够快，方向错了也可以靠继续生成修回来。

实际恰恰相反。速度越快，错误方向堆积得也越快。

7 月 19 日晚上，我让 Claude Code 对 UI 做一次大升级，希望用户打开后会觉得「这个产品很厉害」。它做了新的壳层、六套皮肤和一批新功能。第二天早上我打开，第一反应不是惊喜，而是主链路被破坏了：Chat 不再是中心，Session 的上下文关系变得混乱，页面看起来更丰富，却更难使用。

我最后的决定是：整批撤销。

那次撤销比新增一套功能更重要。它让我确认了几条后来一直坚持的开发规则：

**第一，UI 先讨论结构，再讨论视觉。** 如果信息架构不对，再漂亮的皮肤也只是把错误包装得更精致。

**第二，涉及关键交互，先做可点击的高保真 Mock。** 我确认方向后再进入真实代码，避免 Agent 根据一句抽象描述自由发挥。

**第三，测试通过不等于产品正确。** 自动化可以证明按钮能点、状态能保存、页面没有报错，但不能证明这个入口有存在的必要。

**第四，必须在真实 Electron 界面里验收。** 我要求同时看正常桌面宽度和窄窗口，检查非空渲染、控制台错误、关键路径和截图。很多割裂、遮挡、跳转和焦点问题，只会在真实表面出现。

**第五，关键改动由我保留最后的提交权。** 很多会话里，我都会明确说「先不要 commit，我来验收」。Agent 可以大胆实现，但是否留下，仍然需要一个清晰的人类决策点。

后来，代码清理也遵循同样的原则。我们删掉死代码、幽灵依赖和重复逻辑，优化流式时间线的重算，补齐架构边界检查，但有一条红线：不能因为「clean」而丢失那些已经反复打磨过的用户行为。

## 我怎么让 Claude Code 和 Codex 一起工作

这十天里，两者并没有固定成「一个写代码、一个做产品」的死角色，但我逐渐形成了一套比较稳定的协作方法。

Claude Code 更常承担长链路实现：阅读交接文档、完成一个里程碑、跑测试、修回归、留下实施状态。Codex 更常被我拉来做另一种视角：站在用户角度真实体验、研究类似产品、提出本质不同的方案、质疑一个功能是否真的必要，或者对已经完成的实现做反向审查。

有时顺序也会反过来。Codex 会直接实现一个被确认的设计，Claude Code 再从仓库一致性和真实运行角度补齐。重要的不是给模型贴职位标签，而是避免让同一个上下文既提出方案、又实现方案、最后还自己宣布验收通过。

我用四样东西维持跨 session 的连续性：

- `HANDOFF` 和 `TODO` 记录下一段会话从哪里接；
- ADR 记录为什么做这个决定，以及它替代了什么；
- `IMPLEMENTATION_STATUS` 记录真实跑过哪些测试、哪些失败是已知问题；
- Git 提交把一个可以独立解释、独立回滚的结果固定下来。

这套方法看起来比「一句话让 AI 写完」麻烦，却让我敢于让 Agent 跑更长的任务。因为上下文会结束，模型会变，甚至实现会整批撤销，但决策和证据不会只存在于某个聊天窗口里。

## 十天后，我得到的还不是一个完成品

截至写这篇文章时，Charter 已经发布了 `v1.0.0-beta.2`。

它可以在同一个 Composer 里启动受管 Agent、Claude Code 或 Codex；可以把项目、对话、文件、终端、预览、验证和审查放进同一个 Session；可以用独立 worktree 隔离编码任务；可以管理上下文、Memory 和 Skill；也可以让一个 Session 指挥一组可见的 worker。

M10 做了崩溃恢复和可靠性，M11 做了安全、性能、隐私和无障碍硬化，M12 建起了 macOS、Windows、Linux 的 Beta 发布流水线，并附带校验和、产物清单和 SBOM。

但我仍然不会把它称为 Stable。现在的公开包没有付费代码签名证书，系统会给出安全警告；自动更新、签名和更完整的真实用户验证也还没有结束。

这反而是我希望 Charter 保留的一种诚实：通过了什么，就展示什么；没有证明的，不靠一句「应该没问题」跳过去。

## 最后：我开始重新理解「做产品」

以前我会把做产品理解成：先想清楚功能，再把它们实现出来。

这十天之后，我更愿意把它理解成一个不断缩小误差的过程。

AI 极大地提高了实现速度，也让「做错但做得很完整」变得前所未有地容易。真正稀缺的，不再只是把代码写出来的能力，而是判断什么值得存在、什么应该被删掉、什么时候必须停下来回到用户主链路。

我仍然会让 Agent 在我睡觉时工作，也仍然会把大段实现交给它们。但我不再把「醒来看到很多代码」当成成功标准。

我更在意的是：醒来之后，我能不能看懂它做了什么；证据是否足够；我是否能继续追问；如果方向错了，能不能安全地撤销；如果同时有多个 Agent 在工作，我能不能随时找到、打断和接管其中任何一个。

Charter 这个名字，本意是「章程」或「委托契约」。现在回头看，它也越来越像这个产品真正的核心：

**我把目标交给 Agent，但目标、边界、证据和最后的决定，仍然握在我手里。**

这不是对 Agent 的不信任。

恰恰是因为它们越来越强，我才更需要一套配得上这种能力的工作方式。
